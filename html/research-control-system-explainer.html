Research-Control System

A technical map of the authority hierarchy, the project-system improvement loop, and the source-backed HTML governance rule: edit the Markdown spec first, regenerate the human-only HTML derivative, then validate.

Generated pages support human understanding. They never become independent scientific, registry, or workflow authority.

Source Authority Ladder

1. Registered .tex

Canonical for physics research and derivational claims.

2. Registries

Canonical for provenance, routing, generated-output tracking, and memory.

3. Registered Markdown

Canonical for front-door docs, guidance, and HTML source specs.

4. Generated derivatives

HTML, PDFs, wiki notes, indexes, and local retrieval layers.

Source-Backed HTML Workflow

1. Source spec

Markdown declares purpose, sources, claim boundary, controls, drilldowns, and capsule schema.

Inspect step

Input: source materials. Output: registered source spec. Validator: required frontmatter and capsule body section.

2. Render HTML

The generated page implements progressive disclosure and marker evidence.

Inspect step

Input: Markdown spec. Output: tracked HTML. Boundary: generated human-only derivative.

3. Register hashes

Bootstrap refreshes source hashes, HTML hashes, and generated registries.

Inspect step

Input: live files. Output: registry rows. Validator: source-basis hash alignment.

4. Validate markers

Lightweight checks confirm declared controls and capsule fields appear in HTML.

Inspect step

Input: spec and HTML. Output: pass/fail. Scope: structural evidence only, not design quality.

5. Human review

Browser QA can inspect design, interaction, readability, and claim discipline.

Inspect step

Input: rendered page. Output: human judgment. Boundary: review does not change source authority.

Project-System Improvement Loop

Premise
Project-system improvement is separate from physics continuation.
Mechanism

Source basis
AGENTS, research-control README, and improve-project-system skill.

Uncertainty
