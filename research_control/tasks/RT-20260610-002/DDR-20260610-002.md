# Director Decision Record DDR-20260610-002

## Current Objective

Enforce project-system gate invariants. First, `project_improvement_signals`
emitted by completion or handoff files are detected structurally from YAML,
not by free-text substring matching. Signal types must be defined in
`registries/PROJECT_IMPROVEMENT_SIGNAL_TYPE_REGISTRY.csv`, and concrete
emitted signals must be represented in
`registries/PROJECT_IMPROVEMENT_SIGNAL_REGISTRY.csv`. Signal severity remains
on the concrete signal instance, not the type registry, because urgency depends
on the transaction context and concrete conflict. Signal rows that leave the
open backlog must include explicit resolution evidence: `resolved_by_job_id`,
`resolution_evidence_path`, and `resolved_at`. For `resolved`, `completed`,
and `closed`, the evidence path must point to a PASS completion record with a
matching `job_id`; for `rejected`, the evidence may instead be a Director
decision record that names the signal and explains the rejection. Signal rows
must not duplicate validator command strings; the referenced completion record
owns command evidence through `command_results`. One bounded AgentJob may
close multiple project-improvement signals only when its `objective` names
every closed signal ID and its completion record lists the same IDs with a
nonblank coherent resolution summary. In that shared-closure case, every signal
row must cite the resolving job's canonical `completion_path` from
`AGENT_JOB_REGISTRY.csv`. Second, every
state-changing project-system AgentJob must include a
`documentation_impact.yaml` record; source-doc-only handling is reserved for
plain documentation edits outside a project-system AgentJob. Third, registered
high/critical project-improvement signals route ahead of current Git-change
work, while registered low/medium signals remain backlog unless no current
project-system action is pending. Fourth, the documentation-impact record must
mechanically cover live source changes, generated derivatives, classifier
reason codes, and required validator commands. Fifth, resolver output is
advisory routing state; checkpoint blocking remains defined by validator
failures and concrete authority-boundary violations, not by future-work
visibility. Completion records need not include fresh resolver snapshots solely
because open high or critical signals exist; resolver snapshots remain optional
handoff context unless the AgentJob sets `resolves_signal_routing: true`.
Routing-resolution completions must preserve repo-relative
`resolver_snapshots.before` and `resolver_snapshots.after` paths to JSON
output from `resolve_project_improvement.py --json` and include nonblank
`routing_delta_summary`.
